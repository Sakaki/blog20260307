# 0005. PC 画面での目次サイドバー表示

Date: 2026-03-15
Status: Proposed

## Context

現在の目次（TableOfContents）は記事ヘッダーと本文の間にインラインで表示されている。
PC の広い画面では左右に余白が生じており、目次を右側のサイドバーに配置することで
画面スペースを有効活用し、読者が記事を読みながら常に目次を参照できるようにしたい。

モバイルでは従来のインライン折りたたみ目次をそのまま維持する。

## Decision

### 実装方針

Tailwind CSS のレスポンシブブレークポイント（`lg:`）を使い、
**JavaScript を一切使わず** CSS のみでレイアウトを切り替える。

| 変更種別 | ファイル                                      | 概要                                                         |
| -------- | --------------------------------------------- | ------------------------------------------------------------ |
| 変更     | `src/layouts/BaseLayout.astro`                | `maxWidth` prop を追加し、ページごとに最大幅を切り替え可能に |
| 変更     | `src/layouts/PostLayout.astro`                | 2 カラム CSS Grid レイアウトに変更、目次を 2 箇所に配置      |
| 新規     | `src/components/TableOfContentsSidebar.astro` | サイドバー専用の目次コンポーネント（折りたたみなし）         |

### レイアウト設計

**デスクトップ（lg: 以上）:**

```
┌──────────────────────────────────────────┐
│ Header                                    │
├────────────────────────────┬─────────────┤
│ Article                    │ Sidebar     │
│ ┌────────────────────────┐ │ ┌─────────┐ │
│ │ Title / Date / Tags    │ │ │ 目次    │ │
│ ├────────────────────────┤ │ │ ・h2    │ │
│ │ 本文                   │ │ │  ・h3   │ │
│ │                        │ │ │ ・h2    │ │
│ │                        │ │ │         │ │
│ └────────────────────────┘ │ └─────────┘ │
├────────────────────────────┴─────────────┤
│ Footer                                    │
└──────────────────────────────────────────┘
```

- `max-w-6xl` で 2 カラム分の幅を確保
- CSS Grid: `lg:grid lg:grid-cols-[1fr_16rem] lg:gap-8`
- サイドバーの目次は `sticky top-8` で固定
- インライン目次は `lg:hidden` で非表示

**モバイル（lg: 未満）:**

- 現状と同一（インライン折りたたみ目次）
- サイドバーは `hidden lg:block` で非表示

### 目次を 2 箇所に配置する理由

同じ目次コンポーネントを CSS の表示切り替えで出し分ける。
DOM 上は 2 つの目次が存在するが、常にどちらか片方のみが表示される。

- インライン版: `TableOfContents`（既存、`lg:hidden` を追加）
- サイドバー版: `TableOfContentsSidebar`（新規、`hidden lg:block`）

### TableOfContentsSidebar の設計

- daisyUI の `collapse` は使用しない（サイドバーでは常に展開表示が適切）
- 見出しリストのみをシンプルに描画
- h2/h3 のフィルタリングとインデントは既存の `TableOfContents` と同じロジック
- `sticky top-8` でスクロール追従

### 採用しなかった選択肢

| 選択肢                                     | 不採用の理由                                                             |
| ------------------------------------------ | ------------------------------------------------------------------------ |
| 単一コンポーネントに prop で表示モード切替 | collapse の有無で DOM 構造が大きく異なるため、別コンポーネントの方が明確 |
| JavaScript で目次を動的に移動              | SSG 方針に反する。CSS のみで実現可能                                     |
| サイドバーを左側に配置                     | 記事本文が右寄りになり読みづらい。右サイドバーが一般的なパターン         |
| `max-w-4xl` のまま 2 カラム                | 本文の幅が狭くなりすぎて可読性が低下する                                 |
| 現在位置ハイライト（スクロールスパイ）     | JavaScript が必要であり、SSG 方針と整合しない。将来の拡張として別途検討  |

## Consequences

**メリット:**

- PC 画面の余白を有効活用し、記事を読みながら常に目次を参照できる
- スクロール追従（sticky）により長い記事でも目次が常に可視
- JavaScript 不要で SSG 方針と整合
- モバイルでは既存の挙動が維持され、後方互換性がある
- `BaseLayout` の `maxWidth` prop により、今後他のページでも幅調整が容易

**デメリット・注意点:**

- DOM 上に目次が 2 つ存在するため、HTML サイズがわずかに増加する
- `BaseLayout` の変更は全ページに影響しうるが、デフォルト値を現行と同じ `max-w-4xl` にすることでリスクを最小化
- サイドバー幅 `16rem` は見出しテキストが長い場合に折り返しが発生する可能性がある
